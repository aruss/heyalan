# Milestone M32.2: Team Members Feature Flag

## Summary
Add a frontend-only boolean feature flag named `teamMembers` in `BuyAlan.WebApp` using the existing M32.1 flag system. The flag controls whether team-member UI is exposed in onboarding and admin. `BuyAlan.WebApi` remains unchanged.

## Key Changes
- [x] Extend the existing typed feature-flag registry with `teamMembers`.
- [x] Continue using the existing aggregate `FEATURE_FLAGS` env contract, with entries like `teamMembers=1` and `teamMembers=0`.
- [x] In onboarding, gate the invitations/team-members step UI with `teamMembers`.
- [x] When `teamMembers` is off, onboarding must still remain completable and must not show invite suggestions, invite form, pending invitations, or current members UI.
- [x] In admin, hide the `Members` sidebar entry when `teamMembers` is off.
- [x] Gate the `/admin/settings/members` route itself so direct navigation does not expose live member/invitation management while the flag is off.
- [x] The disabled admin route should render a safe non-interactive fallback message rather than the active management UI.

## Public Interfaces / Types
- [x] Add `teamMembers` to the shared `FeatureFlagKey` and `FeatureFlagSnapshot`.
- [x] Reuse the existing APIs and hooks:
  - [x] `getFeatureFlagSnapshot`
  - [x] `isFeatureEnabled`
  - [x] `FeatureFlagsProvider`
  - [x] `useFeatureFlag`
  - [ ] `useFeatureFlags`
- [x] No backend DTO, API, schema, or generated-client changes.

## Test Plan
- [x] Env parsing:
  - [x] `FEATURE_FLAGS=teamMembers=1` resolves to `true`.
  - [x] `FEATURE_FLAGS=teamMembers=0` resolves to `false`.
  - [x] Missing `teamMembers` falls back to the default.
  - [ ] Onboarding:
  - [ ] Enabled state shows the existing invitations/team-members UI.
  - [ ] Disabled state hides all team-member UI and still allows onboarding completion.
- [ ] Admin:
  - [ ] Enabled state shows the sidebar entry and current members page.
  - [ ] Disabled state hides the sidebar entry.
  - [ ] Disabled state blocks the live members page on direct route visit.
- [ ] Consistency:
  - [ ] Onboarding and admin observe the same evaluated `teamMembers` value.
  - [ ] No hydration mismatch is introduced.

## Assumptions
- [x] `teamMembers` defaults to `true` so current shipped behavior remains visible unless explicitly disabled by deployment config.
- [x] `FEATURE_FLAGS` remains the only runtime configuration surface for frontend flags.
- [x] This is a UI rollout switch only; backend invitation/member endpoints remain available but are out of scope for this milestone.
