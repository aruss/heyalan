# Milestone M32.3: WebApi Feature Flag Service

## Summary
Add a small deployment-scoped feature flag service in `BuyAlan.Configuration` so `BuyAlan.WebApi` can evaluate the same `FEATURE_FLAGS` environment variable already used by the frontend.

- Parse the existing aggregate `FEATURE_FLAGS` env contract in backend code.
- Expose one simple backend API: `IsFeatureEnabled(string featureName)`.
- Keep parsing behavior aligned with the frontend contract.
- Keep the implementation intentionally small: no request-scoped evaluation, no remote config, no new endpoints, and no OpenAPI changes.

## Scope
- [x] `BuyAlan` shared library configuration code.
- [x] `BuyAlan.WebApi` service registration and internal consumption.
- [x] Parsing and default-fallback tests.
- [ ] No frontend rendering or hydration work in this milestone.

## Non-Goals (Out of Scope)
- [ ] No A/B testing, bucketing, stickiness, or user targeting.
- [ ] No feature-flag provider model or request-context evaluation.
- [ ] No admin UI for managing flags.
- [ ] No database, schema, or migration changes.
- [ ] No WebApi endpoint, DTO, schema, or generated-client changes.
- [ ] No replacement of the existing frontend feature-flag implementation.

## User Decisions (Locked)
- [x] The backend API should stay simple and center on `IsFeatureEnabled(string featureName) -> bool`.
- [x] A small list of constants is acceptable for known feature names.
- [x] The backend should read the same `FEATURE_FLAGS` config contract as the frontend.
- [x] The service should live under `BuyAlan/Configuration`.
- [x] No milestone-specific design expansion beyond the simple service is required.

## Findings from Repository Analysis
- [x] Frontend feature flags already use a single aggregate environment variable named `FEATURE_FLAGS`.
- [x] Frontend parsing accepts `1` and `0`, trims whitespace, ignores empty segments, warns on malformed entries, and lets the last duplicate value win.
- [x] Current frontend defaults are `landingPricing = false` and `teamMembers = true`.
- [x] `BuyAlan/Extensions/StringExtensions.cs` already includes `ToBoolean()`, but it accepts broader truthy values such as `true`, `yes`, and `on`, which does not match the frontend feature-flag contract.
- [x] `BuyAlan.WebApi` currently registers shared singleton configuration objects in `AddCoreServices`.
- [x] The repository already has `BuyAlan.Tests` and `BuyAlan.WebApi.Tests` for backend regression coverage.

## Architecture Decisions (Locked)
- [x] Implement a singleton backend service because flag evaluation is deployment-scoped.
- [x] Parse `FEATURE_FLAGS` once from `IConfiguration` during service construction and cache the result in memory.
- [x] Use a simple internal dictionary of parsed values keyed by feature name.
- [x] Keep a small internal defaults map so backend fallback behavior stays aligned with current frontend defaults.
- [x] Do not use `StringExtensions.ToBoolean()` for feature-flag values because it would broaden the accepted config contract beyond the frontend behavior.
- [x] Return `false` for unknown feature names instead of throwing.
- [x] Keep warnings server-side only and avoid logging sensitive configuration payloads.

## Public Interfaces / Contracts
- [x] Add `IFeatureFlagService` in `BuyAlan.Configuration` with:
  - [x] `bool IsFeatureEnabled(string featureName)`
- [x] Add a constants holder for known feature names:
  - [x] `landingPricing`
  - [x] `teamMembers`
- [ ] Reuse the existing aggregate environment contract:
  - [ ] `FEATURE_FLAGS`
  - [ ] `featureName=1;featureTwo=0`

## Gate B: FEATURE_FLAGS Parsing and Evaluation
- [x] Implement `FeatureFlagService` under `BuyAlan/Configuration`.
- [x] Read `FEATURE_FLAGS` from `IConfiguration`.
- [x] Parse semicolon-separated `key=value` segments.
- [x] Trim whitespace around segments, keys, and values.
- [x] Accept `1` as enabled and `0` as disabled.
- [x] Reject other truthy or falsy strings such as `true`, `false`, `yes`, `no`, and `on` so backend behavior stays aligned with frontend parsing.
- [x] Ignore empty segments caused by repeated or trailing separators.
- [x] Warn on malformed segments without exactly one `=`.
- [x] Warn on invalid values other than `1` or `0`.
- [x] Warn on duplicate keys and let the last value win.
- [x] Cache parsed values in memory after construction.
- [x] Return the parsed value when present.
- [x] Return the known default when a known flag is missing from `FEATURE_FLAGS`.
- [x] Return `false` when an unknown feature name is requested.

### Gate B Acceptance Criteria
- [x] Backend parsing behavior matches the established frontend `FEATURE_FLAGS` contract.
- [x] Missing known flags fall back deterministically.
- [x] Invalid config does not crash feature evaluation and instead fails safely.
- [x] Unknown feature-name lookups fail closed.

## Test Cases and Scenarios (Authoritative)
1. **Env parsing**
- [x] `FEATURE_FLAGS=landingPricing=1` resolves `landingPricing` to `true`.
- [x] `FEATURE_FLAGS=landingPricing=0` resolves `landingPricing` to `false`.
- [x] `FEATURE_FLAGS=teamMembers=1` resolves `teamMembers` to `true`.
- [x] `FEATURE_FLAGS=teamMembers=0` resolves `teamMembers` to `false`.
- [x] `FEATURE_FLAGS= landingPricing = 1 ; teamMembers = 0 ;` resolves both values correctly.
- [x] `FEATURE_FLAGS=landingPricing=true` is treated as invalid, logs a warning, and falls back to the known default.
- [x] `FEATURE_FLAGS=teamMembers=yes` is treated as invalid, logs a warning, and falls back to the known default.
- [x] Empty segments are ignored.
- [x] Malformed segments are ignored and log a warning.
- [x] Invalid values are ignored and log a warning.
- [x] Duplicate keys log a warning and the last value wins.

2. **Default fallback**
- [x] Missing `landingPricing` falls back to `false`.
- [x] Missing `teamMembers` falls back to `true`.
- [x] Missing `FEATURE_FLAGS` entirely still returns known defaults.

3. **Unknown feature names**
- [x] `IsFeatureEnabled("doesNotExist")` returns `false`.
- [x] Unknown configured keys do not break known flag resolution.

4. **WebApi registration**
- [x] `IFeatureFlagService` is resolvable from the WebApi service provider.
- [x] The registered service behaves as a singleton.

- [x] Mitigation: log warnings for malformed, duplicate, and invalid entries while failing closed.
- [x] Risk: backend code starts depending on arbitrary string literals.
- [x] Mitigation: provide shared constants for known flags even though the runtime API accepts strings.
