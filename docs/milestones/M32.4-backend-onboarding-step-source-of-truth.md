# Milestone M32.4: Backend Onboarding Steps as Single Source of Truth

## Goal
Make the backend the single source of truth for onboarding steps. Only the backend MUST evaluate the `teamMembers` feature flag for onboarding behavior, and the UI MUST render the onboarding flow strictly from the steps and state returned by the backend.

## Scope
- **Backend (`BuyAlan`)**
  - Refactor onboarding step computation so `teamMembers=false` removes the `invitations` step from backend progression.
  - Ensure backend step normalization, `CurrentStep`, and `CanFinalize` all derive from the same active-step set.
  - Keep the existing onboarding state contract stable where practical.
- **Web API (`BuyAlan.WebApi`)**
  - Align onboarding endpoints with backend step computation so invitation actions are unavailable when `teamMembers=false`.
  - Keep route surface stable; reject disabled invitation actions with deterministic errors instead of removing routes.
- **Tests (`BuyAlan.Tests` / `BuyAlan.WebApi.Tests`)**
  - Add regression coverage for recompute behavior, endpoint behavior, and response consistency when the flag is on and off.

## Non-Goals (Out of Scope)
- Frontend rendering changes in `BuyAlan.WebApp`.
- Changing the `FEATURE_FLAGS` contract or feature-flag service design.
- Database schema or migration changes.
- Removing invitation-related routes from the API surface.
- Changing generated API clients in this milestone.

## User Decisions (Locked)
- [x] The backend MUST be the only place that checks the `teamMembers` flag for onboarding flow.
- [x] The UI SHOULD derive the onboarding flow from backend-provided steps and state instead of checking a frontend feature flag.
- [x] Invitation-specific endpoints SHOULD stay registered but MUST reject requests consistently when the feature is disabled.
- [x] The onboarding response MAY keep the `invitations` payload field for compatibility, but `steps` MUST be the authoritative source for whether the invitations step exists.

## Findings from Repository Analysis
- [x] `SubscriptionOnboardingService` is already the canonical place for onboarding progression:
  - [x] `ComputeStateAsync` derives per-step state, `CurrentStep`, and `CanFinalize`.
  - [x] `RecomputeStateAsync` persists normalized onboarding state after recomputation.
- [x] Onboarding step definitions currently always include `invitations`, and helper methods use the static definition list directly.
- [x] `BuyAlan.WebApi/Onboarding/OnboardingEndpoints.cs` still exposes invitation-specific actions:
  - [x] `POST /onboarding/subscriptions/{subscriptionId}/members/invitations`
  - [x] `POST /onboarding/subscriptions/{subscriptionId}/members/invitations/complete`
- [x] `ToEndpointResultAsync` always builds invitation step enrichment, even though step presence should be backend-driven.
- [x] `teamMembers` already exists in the backend feature flag service and defaults to `true`.

## Architecture Decisions (Locked)
- [x] `SubscriptionOnboardingService` remains the single place that decides which onboarding steps are active.
- [x] Runtime progression logic MUST use an active step list filtered by `teamMembers`.
- [x] When `teamMembers=false`, backend onboarding MUST behave as if `invitations` does not exist.
- [x] `state.steps` is the authoritative signal for step presence; a compatibility `invitations` payload field is not authoritative.
- [x] Disabled invitation actions MUST fail with a dedicated deterministic backend error such as `onboarding_invitations_disabled`.

## Gate A: Backend Step Computation Refactor (Self-Contained)
- [ ] Inject `IFeatureFlagService` into `SubscriptionOnboardingService`.
- [ ] Introduce a private helper that returns the active onboarding step definitions based on `teamMembers`.
- [ ] Refactor `ComputeStateAsync` to use the active step list instead of the static step definition array directly.
- [ ] Refactor step-order helpers to use the active step list:
  - [ ] `GetStepIndex`
  - [ ] `ResolveCurrentStep`
  - [ ] `ResolveNextStepAfterIndex`
  - [ ] `HasCompletedLaterStep`
- [ ] Ensure `BuildStateResult` emits only active steps.
- [ ] Ensure `RecomputeStateAsync` normalizes persisted `CurrentStep` values that point to inactive steps.

### Gate A Acceptance Criteria
- [ ] When `teamMembers=false`, `state.steps` does not contain `invitations`.
- [ ] When `teamMembers=false`, `state.currentStep` is never `invitations`.
- [ ] When persisted state contains `CurrentStep="invitations"` and the flag is off, recompute normalizes to the next active step.
- [ ] When `teamMembers=true`, onboarding progression is unchanged.

### Gate B Acceptance Criteria

### Gate C Acceptance Criteria

### Gate D Acceptance Criteria

## Test Scenarios (Authoritative)
1. **Flag enabled happy path**
   - [ ] `teamMembers=true` returns the existing onboarding steps including `invitations`.
   - [ ] Invitation completion and finalize behavior remain unchanged.
2. **Flag disabled recompute path**
   - [ ] `teamMembers=false` omits `invitations` from `steps`.
   - [ ] `CurrentStep="invitations"` is normalized to the next active step.
   - [ ] `CanFinalize` is `true` once remaining active required steps are complete.
3. **Flag disabled invitation action path**
   - [ ] Invitation creation endpoint returns `onboarding_invitations_disabled`.
   - [ ] Invitation completion endpoint returns `onboarding_invitations_disabled`.
4. **Flag disabled response-shape path**
   - [ ] Onboarding response remains structurally compatible.
   - [ ] Invitation enrichment is empty/default and does not contradict `steps`.

## Risks and Mitigations
- [ ] Risk: helper methods partially keep using the static step list, causing inconsistent indexes and step normalization.
  - [ ] Mitigation: make all progression helpers consume the same active step list.
- [ ] Risk: WebAPI still exposes invitation behavior even after recompute removes the step.
  - [ ] Mitigation: explicitly gate invitation endpoints and endpoint enrichment.
- [ ] Risk: UI keeps relying on `invitations` payload presence instead of `steps`.
  - [ ] Mitigation: document and test that `steps` is the authoritative contract for step existence.
