# Milestone M32.1: Frontend Feature Flags

## Summary
Implement a typed frontend feature flag system in `BuyAlan.WebApp` that works in SSR-rendered routes and client-side code, and leaves a clean extension path for future A/B testing without implementing A/B assignment in this milestone.

- Evaluate flags on the server for SSR-safe rendering.
- Hydrate the evaluated flag snapshot into client code.
- Make the same flag API available across landing, login, onboarding, and admin surfaces.
- Ship one concrete production flag on the landing page to prove the system against a real user-facing behavior.
- Use a code-defined registry with environment-driven values for v1.
- Keep the evaluation and transport layers separate so future request-scoped or user-scoped experimentation can extend the system without changing consumers.

## Scope
- [ ] `BuyAlan.WebApp` only.
- [ ] App Router SSR usage in server layouts and server components.
- [ ] Client-side consumption through a shared provider and hooks.

- [ ] Concrete landing-page flag for pricing visibility and pricing-related navigation visibility.

## Findings from Repository Analysis
- [x] `BuyAlan.WebApp` uses Next.js App Router with separate root layouts for `(landing)`, `login`, `onboarding`, and `admin`.
- [x] There is no single global root layout already shared by all surfaces.
- [x] Server-side request data is already read in layouts via `cookies()` from `next/headers`.
- [x] Client-side shared state already uses provider patterns such as `SessionProvider` and `BreadcrumbProvider`.
- [x] There is currently no shared feature flag or runtime config layer in the frontend.
- [x] SSR-safe flag support must therefore be introduced as a shared pattern that each root layout opts into.

## Architecture Decisions (Locked)
- [x] Define flags in a typed registry under `src/lib/feature-flags`.
- [x] Resolve flag values on the server from environment variables, not directly in client code.
- [x] Hydrate a serializable flag snapshot into a client provider.
- [x] Expose server and client consumption APIs backed by the same snapshot shape.
- [x] Keep provider placement as deep as practical inside each root layout body instead of wrapping unrelated document structure.
- [x] Keep v1 flag values boolean-only.
- [x] Use the landing pricing flag as the authoritative reference implementation for real feature-gated UI.
- [x] Keep internal architecture open for future evaluator inputs such as cookies, user identity, or request context without committing to those inputs now.

## Implementation Plan by Gate

- [x] Create a central typed feature flag registry in `src/lib/feature-flags`.
- [x] Define each flag with:

  - [x] `defaultValue`
  - [x] `description`
  - [x] optional metadata fields that do not affect runtime behavior in v1
- [x] Seed the registry with `landingPricing` as the first production flag.
- [x] Add parsing rules for env values:
  - [x] `1`
  - [x] `0`
- [x] Fall back to registry defaults when env values are missing or invalid.
- [x] Emit a server-side warning for invalid flag values without exposing sensitive runtime config.

### Gate A Acceptance Criteria
- [x] Every flag is declared in one typed registry.
- [x] Unknown flag keys are rejected by TypeScript.
- [x] Missing or invalid env values produce deterministic defaults.
- [x] Client code does not read raw env vars directly.
- [x] `landingPricing` resolves from the aggregate `FEATURE_FLAGS` payload.

## Gate A.5: Aggregate Environment Variable Contract
- [x] Replace per-flag env lookup with a single `FEATURE_FLAGS` env var.
- [x] Parse semicolon-separated `key=value` pairs where keys are `FeatureFlagKey` values.
- [x] Accept `1` for enabled and `0` for disabled values.
- [x] Trim whitespace around entries, keys, and values.
- [x] Ignore empty segments caused by trailing or repeated separators.
- [x] Fall back to registry defaults when a known flag is missing from `FEATURE_FLAGS`.
- [x] Emit a server-side warning for:
  - [x] unknown flag keys
  - [x] duplicate keys
  - [x] invalid values
  - [x] malformed segments without exactly one `=`

### Gate A.5 Acceptance Criteria
- [x] `FEATURE_FLAGS` is the only supported runtime env contract for frontend flags.
- [x] Known flags resolve from `FEATURE_FLAGS` without changing the snapshot or hook APIs.
- [x] Invalid or unknown entries do not crash SSR and instead fall back safely.
- [x] Duplicate entries are handled deterministically.

## Test Cases and Scenarios (Authoritative)
1. **Env parsing**
- [ ] `FEATURE_FLAGS=landingPricing=1` resolves `landingPricing` to `true`.
- [ ] `FEATURE_FLAGS=landingPricing=0` resolves `landingPricing` to `false`.
- [ ] `FEATURE_FLAGS= landingPricing = 1 ;` resolves correctly despite whitespace and trailing separators.
- [ ] Missing env uses the registry default.
- [ ] Missing known key in `FEATURE_FLAGS` uses the registry default.
- [ ] Invalid value in `FEATURE_FLAGS` uses the registry default and logs a warning.
- [ ] Unknown key in `FEATURE_FLAGS` is ignored and logs a warning.
- [ ] Duplicate keys in `FEATURE_FLAGS` log a warning and the last value wins.
- [ ] Malformed segments in `FEATURE_FLAGS` are ignored and log a warning.

2. **Server evaluation**
- [ ] A server helper can read a known flag by key.
- [ ] The generated snapshot contains every registered flag exactly once.

3. **Client provider**
- [ ] `FeatureFlagsProvider` exposes hydrated values to descendants.
- [ ] `useFeatureFlag` returns the expected boolean for a known key.
- [ ] Hook usage outside the provider fails clearly.

4. **SSR + hydration consistency**
- [ ] The landing page omits the pricing section on the server when `landingPricing` is off.
- [ ] The same route hydrates without mismatch when landing navigation reads the same flag.
- [ ] Enabled and disabled landing states both render consistently.

5. **Cross-surface availability**
- [ ] Flag wiring works in at least one route under landing/login and one route under onboarding/admin.
- [ ] Root layout integration does not regress existing providers or route behavior.

6. **Concrete landing pricing behavior**
- [ ] Enabled state renders the pricing section and pricing-related landing navigation normally.
- [ ] Disabled state removes the landing pricing section.
- [ ] Disabled state removes all landing links to `#pricing`.
- [ ] Disabled state removes the landing `Start Free Trial` menu item.

## Risks and Mitigations
- [ ] Risk: duplicated wiring across multiple root layouts drifts over time.
- [ ] Mitigation: keep one shared snapshot/provider helper pattern and apply it uniformly.
- [ ] Risk: server-only env reads accidentally leak into client bundles.
- [ ] Mitigation: isolate env parsing and snapshot building in server-only helpers.
- [ ] Risk: aggregate env payloads become harder to reason about during deployment changes.
- [ ] Mitigation: keep parsing strict, warnings specific, and the payload format small and deterministic.
- [ ] Risk: hydration mismatch when SSR and client code evaluate differently.
- [ ] Mitigation: client code reads only the server-generated snapshot in v1.
- [ ] Risk: hiding the pricing section but leaving `#pricing` links creates broken landing navigation.
- [ ] Mitigation: treat pricing content and pricing-related menu items as one flag-owned behavior.
- [ ] Risk: future A/B work becomes constrained by a boolean-only API.
- [ ] Mitigation: keep the evaluator and transport abstractions separate even though v1 returns booleans.
