# M35.1 Logger Simplification: Remove Pino, Direct OTEL Logs

## Summary
Replace the WebApp server logger with a small server-only logger built on `@opentelemetry` logs packages. Remove `pino` and `pino-opentelemetry-transport` completely from the WebApp codebase and dependency graph, send logs directly to the OTEL collector over gRPC, and make both configuration and logger method names follow `.NET` terminology.

The new logger should be minimal and explicit:
- config uses `.NET` log levels: `Trace`, `Debug`, `Information`, `Warning`, `Error`, `Critical`, `None`
- logger methods use `.NET` naming as well
- no worker transport, no bundler-specific Pino handling, no structured stdout logger dependency
- OTEL export is best-effort and must never prevent app startup
- after implementation there must be no `pino` packages and no `pino` references left anywhere in `BuyAlan.WebApp`

## Key Changes
- Replace the current logger implementation in the WebApp with a custom logger module that:
  - exposes `createLogger(bindings)` for scoped loggers
  - exposes methods `trace`, `debug`, `information`, `warning`, `error`, `critical`
  - optionally exposes a generic `log(level, message, fields?)` plus `isEnabled(level)`
  - keeps `serializeError`, flush, and shutdown support for fatal-path handling
- Remove `pino` completely from the WebApp implementation:
  - stop importing `pino` types in `telemetry-config.ts`
  - remove any `pino` compatibility aliases, comments, helpers, and test references
  - resolve `LOG_LEVEL` to a new internal `.NET` log-level enum/string union
  - keep case-insensitive parsing and safe fallback to `Information`
  - keep one startup warning for invalid `LOG_LEVEL`
- Build an OTEL logs bridge with `@opentelemetry/api-logs`, `@opentelemetry/sdk-logs`, and `@opentelemetry/exporter-logs-otlp-grpc`:
  - create a singleton `LoggerProvider`
  - configure it with shared resource attributes already used by traces/metrics
  - use `BatchLogRecordProcessor`
  - export directly to the Aspire OTEL collector via gRPC
  - support only gRPC for logs; if another OTLP logs protocol is configured, emit one warning and disable OTEL log export rather than guessing
- Keep env precedence already established for logs:
  - `OTEL_EXPORTER_OTLP_LOGS_ENDPOINT` overrides `OTEL_EXPORTER_OTLP_ENDPOINT`
  - `OTEL_EXPORTER_OTLP_LOGS_PROTOCOL` overrides `OTEL_EXPORTER_OTLP_PROTOCOL`
  - `OTEL_SERVICE_NAME`, `APP_VERSION`, and `NODE_ENV` remain the resource sources
- Update existing call sites to the new API:
  - startup logging in `instrumentation.otel.ts`
  - server error hooks
  - feature-flag warnings
  - health probe logging
  - temporary landing-page test logs if they are still needed
- Keep emergency fallback logging via `console.error` only for logger/bootstrap failure paths; do not keep structured stdout as a first-class sink
- Update `package.json` dependencies:
  - remove `pino` and `pino-opentelemetry-transport`
  - add direct OTEL logs dependencies explicitly if they are currently only transitive
  - before editing, run `npm view` for the OTEL logs packages and pin compatible versions intentionally
- Remove Pino-style logger naming from the codebase:
  - replace `info`, `warn`, and `fatal` logger call sites with `.NET`-style method names
  - do not leave compatibility wrappers that preserve Pino-style method names

## Public Interfaces / Behavior
- `LOG_LEVEL` contract becomes fully `.NET`-style:
  - accepted values: `Trace`, `Debug`, `Information`, `Warning`, `Error`, `Critical`, `None`
  - matching remains case-insensitive
  - blank/missing defaults to `Information`
  - invalid values fall back to `Information` and generate one warning
- Logger API uses only `.NET`-style methods.
  - current `info`, `warn`, and `fatal` call sites are updated to `information`, `warning`, and `critical`
  - no Pino-style method aliases remain
- OTEL log export behavior is explicit:
  - direct OTLP/gRPC only
  - no `/v1/logs` suffix for gRPC endpoints
  - no Pino transport worker
  - no stdout JSON log sink except emergency fallback paths

## Test Plan
- Log-level resolution:
  - each `.NET` value resolves correctly
  - case-insensitive inputs work
  - blank/missing falls back to `Information`
  - invalid input warns once and falls back safely
- Logger behavior:
  - scoped loggers retain bound attributes
  - disabled levels do not emit records
  - `Critical` and `None` behave correctly
  - `serializeError` output remains stable for `Error`, strings, primitives, and non-Error throws
- OTEL bridge:
  - gRPC exporter initializes from logs-specific env vars and global fallback env vars
  - unsupported logs protocol disables OTEL log export with a warning instead of crashing
  - resource attributes match traces/metrics
  - flush and shutdown complete without throwing when exporter is unavailable
- App scenarios:
  - startup debug event is emitted only when `LOG_LEVEL=Debug` or `Trace`
  - `/health?logProbe=1` emits one `Information` record with the expected safe fields
  - unhandled rejection and uncaught exception paths still log and flush
  - landing-page/server route log calls reach the OTEL backend in Aspire
- Removal validation:
  - `BuyAlan.WebApp/package.json` no longer contains `pino` or `pino-opentelemetry-transport`
  - repo search under `BuyAlan.WebApp` returns no remaining `pino` references
- Validation commands should remain non-scripted where possible, using targeted `node --test` and `eslint` invocation rather than package scripts

## Assumptions and Defaults
- The logger remains server-only; no client bundle logging bridge is added.
- The primary target is Aspire’s OTEL collector, not stdout log viewing.
- OTEL logs protocol support is intentionally limited to gRPC for now.
- Method naming will be standardized on `.NET` terminology rather than preserving Pino-style aliases.
- Existing milestone `M35.1` should be updated to reflect the direct-OTEL custom logger approach replacing the earlier Pino-based design.
